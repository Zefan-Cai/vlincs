VLINCS global ID

Turning Open-World ReID into a VLINCS pipeline component

We stopped treating the task as "force every tracklet into a class." After tracklet generation, we add a no-anchor identity-resolution layer: gather evidence, repair the identity graph, then emit global IDs.

Core idea

From forced output to evidence-based resolution

Previous pipelinevideo -> detection/tracking -> tracklets -> feature/search -> forced global_id submission
Added layertracklets -> evidence graph -> candidate repairs -> graph resolution -> calibrated delivery

The global ID is no longer just the direct output of search or classification. It becomes an identity decision supported by evidence, conflict checks, candidate repairs, and delivery validation.

New subsection

Old search/classifier vs new identity resolution

The old system asks: "Which existing ID looks most similar?" The new system asks: "Given all evidence and counter-evidence, should these tracklets belong to the same identity?"

Search/classifierTracklet-level nearest-neighbor matching or classification. At the end, every tracklet must receive a global ID.
Evidence-based resolutionComponent-level reasoning over evidence, counter-evidence, and side effects. Only verified identity edits are committed.
Before

How the previous search/classifier worked

Here, "classifier" does not mean a fixed trained class head. It is more accurately an online gallery matcher: it reads tracklets and maintains a live identity gallery.

1. Convert tracklet to vector
  • Input: multi-frame bbox/crop evidence, camera, and time span.
  • The frames are pooled into one appearance vector for the tracklet.
2. Search the gallery
  • Each gallery ID stores exemplars or a centroid.
  • The new tracklet vector is compared to existing identity vectors by cosine similarity.
  • The matcher can use centroid, max exemplar, or FAISS top-k retrieval.
3. Match or expand
  • If the top ID clears the similarity threshold and no same-frame or travel-time cannot-link fires, the tracklet is matched to that ID.
  • If no candidate qualifies, the system expands by creating a new global ID.
4. Emit predicted_global_id
  • After match or expand, the chosen ID is written into the submission.
  • A later global resolve step may recluster, but the output is still forced: every tracklet receives an ID.
Concrete example

What happens when one tracklet arrives

Step 1Tracklet T is pooled into vector v_T.
Step 2The system compares v_T to the gallery: ID 41 is most similar, ID 83 is second, ID 350 is third.
Step 3If ID 41 has no cannot-link and clears the threshold, T is written as predicted_global_id = 41.
Step 4If no candidate is similar enough, the system creates a new ID. That new ID enters the gallery and affects later tracklets.

The old method is basically "choose the nearest usable identity." It does not deeply ask whether the decision has enough evidence, has counterexamples, or harms other components.

After

How the current identity decision works

Evidence graph
  • Nodes are tracklets and identity components, not single frames.
  • Evidence includes visual features, camera, time, bbox geometry, tracklet length, and the current assignment.
Conflict signals
  • We look for inconsistent islands inside a component.
  • Time, camera, similarity, and component context provide counter-evidence and side-effect risk.
Candidate repairs
  • We propose local edits, such as moving a few tracklets from a source component to a target component.
  • Small repairs are preferred over whole-component hard merges.
Calibrated delivery
  • We score the direct candidate first, then run the real delivery path.
  • Only candidates that improve end-to-end IDF1 are promoted into the best state.
Weak supervision

Where weak positives come from

A weak positive is not ground truth and not a human anchor. It is weak evidence that two tracklets are likely the same person, used to train and rank a pair-link model.

Resolver agreement
  • Run no-GT resolvers using visual features and temporal proximity.
  • If multiple resolver settings place two tracklets in the same pseudo component, that pair gets positive votes.
Online gallery agreement
  • If the old gallery also assigns both tracklets to the same non-singleton ID, we treat it as independent agreement.
  • This is more reliable than visual similarity alone.
Strong visual pseudo
  • If visual similarity is very high and the pseudo resolver also supports the link, the pair becomes a strong weak positive.
  • This captures obvious same-person fragments across time or cameras.
Consensus positive
  • If several weak signals vote together and similarity clears the threshold, the pair can be accepted.
  • The key rule is multi-evidence agreement, not a single score.
Weak supervision

Where hard negatives come from

A hard negative is weak counter-evidence that two tracklets should not share an identity. It prevents the model from learning only how to merge similar-looking people.

Physical cannot-link
  • If two tracklets overlap in the same video/camera and their boxes are not duplicate boxes, they are almost certainly two people.
  • Impossible cross-camera timing or movement can also block a merge.
No vote plus low similarity
  • If pseudo resolvers give no same-person vote, the old gallery uses different IDs, and visual similarity is low, the pair becomes a negative.
  • This provides ordinary background negatives.
Random background negatives
  • We sample unrelated tracklet pairs outside the candidate set.
  • This stabilizes training so the model does not see only boundary cases.
Historically bad repairs
  • Candidate families that were full-scored and got worse are recorded.
  • The reviewer later deprioritizes those families to avoid repeated failed edits.
How weak labels are used

They are training signals, not final answers

Pair constructionBuild tracklet pairs and compute visual similarity, camera/video relation, time gap, bbox size similarity, tracklet length, and detector confidence.
Weak labelUse weak positives and hard negatives to assign 1/0 pseudo labels to those pairs.
Pair-link modelTrain a pair-link calibrator that predicts whether two tracklets should connect within the same identity component.
Candidate repair rankingThe model does not directly output the final global ID. It ranks which local component repairs are worth full-score validation.
How we did it

Find mistakes, repair small groups, accept only proven gains

1. Find candidate mistakes

Treat tracklets as graph nodes and automatically search for false splits and residual identity islands.

2. Repair only local groups

Do not hard-merge large components. Move only a few tracklets when their evidence agrees.

3. Validate delivery

Every repair must pass real submission scoring. It enters the best state only when end-to-end IDF1 improves.

No-anchor evidence

No human anchors and no GT as production evidence

Candidate generation uses only the current assignment, tracklet metadata, camera/time information, and tracklet-level visual features such as weakmetric/OSNet, DINOv2, and SigLIP.

Ground truth is used only after a candidate is materialized, for evaluator scoring and ablation accounting. It is not used to generate global IDs or train production evidence.

Takeaway

The improvement is in the identity layer

The detector and tracker were not retrained. The gain comes from open-world identity resolution after tracklets: represent identity as a graph, repair small evidence-supported errors, and use delivery scoring to reject repairs that only look plausible.
